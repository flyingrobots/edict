# v0.4.0-alpha.1 Release Notes

Target date: 2026-07-29

Status: publish-ready prerelease notes for `v0.4.0-alpha.1`.

## Release Type

`v0.4.0-alpha.1` is the target-profile and lowerability alpha. It adds typed
runtime-neutral target-profile conformance checks, typed v1 lowerability checks
for direct-only target support, and typed participant-neutral contract bundle
manifest validation.

This is not an admission-stack release and does not execute target lowerers.

## Version Policy

- `crates/edict-syntax` is versioned as `0.4.0-alpha.1`; its package remains
  `publish = false`.
- `xtask` remains an internal workspace helper at `0.0.0`.
- Versions currently track repository/specification maturity, not crates.io API
  stability.

## Included

- Typed v1 target-profile manifest conformance in `edict_syntax`, including
  runtime-neutral Echo and KV profile shapes.
- Digest-locked target-profile component validation, Core ABI validation,
  canonical CBOR profile validation, deferred lawpack-adapter ABI rejection, and
  atomic application doctrine validation.
- Typed `LoweringRequirements`, `TargetProfileFacts`, and `check_lowerability`
  checks for native support, exactly one direct adapter, and unsupported
  classifications.
- Rejection of ambiguous native support, ambiguous direct adapters, floating
  adapter references, chained/composite adapter claims, and guard-incompatible
  support facts in the v1 lowerability contract.
- Typed v1 contract-bundle manifest validation for participant-neutral bundle
  artifacts, release-only provenance inputs, canonicalization-profile pinning,
  logical source path validation, optional external assurance evidence binding,
  and explicit rejection of admission artifacts from the bundle.
- Topic shelves and verification matrices for target profiles, lowerability, and
  contract bundles.
- Release-process policy metadata for the v0.4 alpha boundary.

## Explicit Non-Goals

The `v0.4.0-alpha.1` release explicitly does not include:

- crates.io publication;
- target lowerer execution;
- Target IR generation from Core;
- file-backed target-profile or contract-bundle manifest loading;
- canonical-CBOR encode/decode helpers for contract bundle manifests;
- recomputation of semantic or release bundle digests from canonical preimages;
- admission requests, receipts, policies, catalogs, participant descriptors, or
  signatures;
- `edict explain lowerability` or another CLI surface;
- composite profile validation or chained adapter search.

## Required Verification

Before tagging:

- `cargo xtask verify` must pass on the release commit.
- GitHub Actions on `main` must pass for MSRV Rust 1.85.0 and stable Rust.
- Issues #1, #5, and #39 must be closed or explicitly deferred.
- The `v0.4.0-alpha.1` milestone must have no open release-blocking issues.
- The release notes must distinguish target-profile/lowerability/bundle
  validation from target-lowering execution and admission tooling.

## Tagging Plan

After the release-prep PR lands and the `main` release commit is verified:

```text
git tag -a v0.4.0-alpha.1 -m "v0.4.0-alpha.1"
git push origin v0.4.0-alpha.1
```

Pushing that tag to GitHub triggers the release workflow. The workflow verifies
that the tag target is reachable from `origin/main`, loads these release notes
from `docs/releases/v0.4.0-alpha.1.md`, and publishes the GitHub Release as a
prerelease. It does not publish crates.
