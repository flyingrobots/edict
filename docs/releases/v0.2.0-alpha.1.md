# v0.2.0-alpha.1 Release Notes

Target date: 2026-07-01

Status: published as a GitHub prerelease from the signed `v0.2.0-alpha.1` tag
on 2026-06-21.

## Release Type

`v0.2.0-alpha.1` is the Core semantic model and normative schema alpha. It
freezes Core meaning and schema shape enough for downstream compiler-spine work
to begin from a stable contract.

This is not a compiler-spine, target-lowering, or admission-stack release.

## Version Policy

- `crates/edict-syntax` is versioned as `0.2.0-alpha.1`; its package remains
  `publish = false`.
- `xtask` remains an internal workspace helper at `0.0.0`.
- Versions currently track repository/specification maturity, not crates.io API
  stability.

## Included

- Normative `edict.core/v1` CDDL schema in `docs/abi/edict-core.cddl`.
- Core semantic model coverage for modules, imports, types, values, expressions,
  predicates, input constraints, intents, blocks, nodes, local references, and
  pure Core helper bodies.
- `docs/topics/core-ir/` current-truth chapter and verification matrix.
- Accepted/rejected Core schema-shape fixtures under `fixtures/core/schema/`.
- `xtask` regressions proving required Core declarations, no byte/hash freeze
  fields, and schema-shape fixture acceptance/rejection.
- Release-process topic shelf and structured release-tag recovery policy.
- Repo-local `AGENTS.md` policy for Think, git rules, topic shelves, and local
  verification.

## Explicit Non-Goals

The `v0.2.0-alpha.1` release explicitly does not include:

- crates.io publication;
- source-to-Core lowering;
- a compiler CLI;
- import resolution or resolved type checking;
- a canonical Core encoder;
- golden Core bytes;
- exact Core digests;
- a full CDDL instance-validation harness;
- target profile conformance fixtures;
- target lowerers;
- bundle/admission tooling.

## Required Verification

Before tagging:

- `cargo xtask verify` must pass on the release commit.
- GitHub Actions on `main` must pass for MSRV Rust 1.85.0 and stable Rust.
- Issues #3, #19, and #28 must be closed or explicitly deferred.
- The `v0.2.0-alpha.1` milestone must have no open release-blocking issues.

## Tagging Plan

After the release-prep PR lands and the `main` release commit is verified:

```text
git tag -a v0.2.0-alpha.1 -m "v0.2.0-alpha.1"
git push origin v0.2.0-alpha.1
```

Pushing that tag to GitHub triggers the release workflow. The workflow verifies
that the tag target is reachable from `origin/main`, loads these release notes
from `docs/releases/v0.2.0-alpha.1.md`, and publishes the GitHub Release as a
prerelease. It does not publish crates.
