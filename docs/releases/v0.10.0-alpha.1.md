# v0.10.0-alpha.1 Release Notes

Target date: 2026-10-21

Status: publish-ready release notes for `v0.10.0-alpha.1`. Publication evidence
is added after the immutable tag is created.

## Release Type

`v0.10.0-alpha.1` is the first public command-line alpha. It exposes the
`edict check` workflow over a JSONL-only stdin/stdout/stderr contract, with
checked-in JSON Schemas for every CLI stream record family and a golden fixture
corpus replayed end-to-end through the binary. It does not add compile, lower,
explain, bundle, or admission command workflows, a human-pretty output mode, a
language server, or crates.io publication.

## Release Thesis

`v0.10.0-alpha.1` proves Edict's first public boundary: an `edict` binary that
reads JSONL compiler-settings and compiler-input records on stdin and emits only
JSONL success, diagnostic, and terminal status records on stdout and stderr,
checking inline source, file paths, directories, ordered path lists, and glob
patterns deterministically. Every record family is pinned to a checked-in JSON
Schema, and a golden fixture corpus replays the binary byte-for-byte so the
public contract cannot drift silently.

The release deliberately excludes compile, lower, explain, bundle, and admission
command workflows, human-pretty output, an embedded JSON Schema validation
engine, a language server, marketplace packaging, participant policy execution,
and crates.io publication.

## Included

- A public `edict` binary (`crates/edict-cli`) implementing the `check`
  operation over a JSONL-only stdin/stdout/stderr contract.
- JSONL compiler-settings and compiler-input records on stdin, accepting inline
  source, file paths, directories, ordered path lists, and glob patterns with
  deterministic, sorted expansion.
- JSONL-only stdout and stderr records for success results, structured parse,
  semantic, and CLI diagnostics, and terminal status events.
- Checked-in JSON Schema contract artifacts for all five CLI stream record
  families: `edict.compiler.settings/v1`, `edict.compiler.input/v1`,
  `edict.cli.check-result/v1`, `edict.cli.diagnostic/v1`, and
  `edict.cli.event/v1`.
- Schema guards that pin each record's identifier, required fields, mutually
  exclusive compiler-input kinds, and the terminal `status`/`exitCode` coupling,
  with the runtime schema identifiers hoisted into `edict_cli` as a single
  source of truth.
- Stable CLI diagnostic `kind` codes derived from explicit, exhaustive
  `ParseErrorKind::code` and `SemanticErrorKind::code` mappings instead of
  `Debug` formatting.
- A golden CLI fixture corpus under `fixtures/cli/`, replayed end-to-end through
  the binary by `golden_cli_fixtures_replay_exactly`, matching stdout, stderr,
  and exit code byte-for-byte across success, parse rejection, semantic
  rejection, CLI-input rejection, GraphQL-shape-importing source, and the
  deterministic path, directory, path-list, and glob expansion paths.
- CLI topic-shelf coverage and the `fixtures/cli/` corpus documentation.
- Post-`v0.9.0-alpha.1` release-process evidence that marks the first Target IR
  alpha as published without changing the immutable release tag.

## Explicit Non-Goals

The `v0.10.0-alpha.1` release explicitly does not include:

- crates.io publication;
- compile, lower, explain, bundle, or admission command workflows;
- a human-pretty CLI output mode;
- an embedded JSON Schema validation engine in the CLI;
- a language server or editor diagnostic transport;
- marketplace or distribution packaging;
- participant admission policy execution.

## Tagging Plan

The auto-release-tag workflow derives the tag from the merged release branch:

```text
release/v0.10.0-alpha.1-prep -> v0.10.0-alpha.1
```

After the release-prep PR merges and `main` CI succeeds, automation creates the
annotated `v0.10.0-alpha.1` tag on the verified `main` merge commit and
dispatches the Release workflow. The Release workflow verifies that the tag
target is reachable from `origin/main`, publishes the GitHub Release as a
prerelease, and closes the matching milestone after confirming zero open issues.
It does not publish crates.

If automation fails after a valid tag exists, do not move, delete, recreate, or
force-push the tag. Use the non-mutating recovery paths in the release-process
runbook.
