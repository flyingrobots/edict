# v0.3.0-alpha.1 Release Notes

Target date: 2026-07-15

Status: published as a GitHub prerelease from the signed `v0.3.0-alpha.1` tag
targeting `4ea3d993f74490b495fe6e6a9ec2d52f889ccceb`.

## Release Type

`v0.3.0-alpha.1` is the compiler-spine alpha. It connects the source front end
to in-memory Core IR for the initial executable subset, adds the reference
canonical Core encoder, and checks the first reviewed Core golden bytes and
exact digest fixture.

This is not a full compiler, target-lowering, or admission-stack release.

## Version Policy

- `crates/edict-syntax` is versioned as `0.3.0-alpha.1`; its package remains
  `publish = false`.
- `xtask` remains an internal workspace helper at `0.0.0`.
- Versions currently track repository/specification maturity, not crates.io API
  stability.

## Included

- `validate_surface` as the explicit source/surface semantic-validation stage,
  with `validate_module` retained as a compatibility alias.
- Executable compiler-spine APIs in `edict_syntax`: `resolve_module`,
  `type_check`, `lower_core`, and `compile_to_core`.
- Deterministic `CompilerContext` profile and budget facts, making contextual
  compiler knowledge explicit instead of implicit.
- A typed module boundary distinct from source AST and in-memory Core lowering
  for the initial pure local-record subset.
- Reference `edict.canonical-cbor/v1` Core encoding for current in-memory Core
  modules.
- Canonical byte validation through decode and byte-identical re-encoding,
  including stable rejection of malformed or non-canonical CBOR inputs.
- Domain-separated `edict.core.module/v1` SHA-256 digest computation over the
  canonical digest frame.
- Reviewed `bounded-hello` Core golden bytes and exact digest fixture:
  `sha256:f8243875a77aa3cc8f02529e2c8b18e183ed6b02ad5f6fd6d0795c6ad532ce85`.
- `cargo xtask core-goldens --check/--write` for deterministic regeneration of
  the reviewed Core golden artifacts.
- Topic-shelf updates for semantic validation, compiler spine, Core IR, testing
  workflow, documentation standards, and release process.
- Release runbook and structured release policy metadata for repeatable alpha
  releases.

## Explicit Non-Goals

The `v0.3.0-alpha.1` release explicitly does not include:

- crates.io publication;
- full source-language lowering beyond the initial pure local-record subset;
- a compiler CLI;
- deferred minimal-v1 syntax such as `fn`/`const`, `record` effects, and
  list/map/unit expression literals;
- full CDDL instance validation;
- target profile conformance fixtures;
- target lowerers;
- bundle/admission tooling.

## Required Verification

Before tagging:

- `cargo xtask verify` must pass on the release commit.
- GitHub Actions on `main` must pass for MSRV Rust 1.85.0 and stable Rust.
- Issues #10, #20, #21, #22, and #35 must be closed or explicitly deferred.
- The `v0.3.0-alpha.1` milestone must have no open release-blocking issues.
- The checked-in `bounded-hello` Core golden bytes and digest must match the
  executable encoder and digest path through `cargo xtask core-goldens --check`.

## Tagging Plan

After the release-prep PR lands and the `main` release commit is verified:

```text
git tag -a v0.3.0-alpha.1 -m "v0.3.0-alpha.1"
git push origin v0.3.0-alpha.1
```

Pushing that tag to GitHub triggers the release workflow. The workflow verifies
that the tag target is reachable from `origin/main`, loads these release notes
from `docs/releases/v0.3.0-alpha.1.md`, and publishes the GitHub Release as a
prerelease. It does not publish crates.
